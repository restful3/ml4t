# study-deck-v1 디자인 규칙 · Chapter 1 완성본 기준

이 템플릿은 ConnectBrick 발표자료의 정보 위계와 발표 UX를 `ML4T study`에 맞게 옮기고, Chapter 1에서 검증한 28장 리포트 파생 흐름과 시각 컴포넌트를 공용화한 1280×720 HTML 덱이다. 로고·고객명·개인 브랜딩은 포함하지 않는다.

- 완성 기준: 아직 완성 회차가 없다. 첫 완성 회차 `docs/studies/machine-trading/presentations/<session-slug>/index.html`가 기준 예시가 되며, 그 전까지는 이 문서와 청사진의 완성 조건을 따른다.
- 공통 설계 청사진: `agent-support/templates/STUDY_SESSION_BLUEPRINT.md`

한 회차의 기본 산출물은 `study-report-v1` 상세 리포트와 이 발표자료의 쌍이다. 원자료에서 리포트를 먼저 완성·검증하고 이 덱은 그 리포트에서만 파생한다. 리포트는 사후 학습을 위한 맥락과 근거를 보존하고, 슬라이드는 같은 주장·용어·표·도형을 발표 흐름과 핵심 메시지로 압축한다. 슬라이드에만 존재하는 새 주장을 만들지 않는다.

## 표준 발표 흐름

50분 발표 기준 18–30장을 권장한다. 기본 HTML은 아래 22장 구조 청사진을 제공하며, 리포트의 필수 시각자료가 많으면 Chapter 1처럼 28장 안팎으로 나눌 수 있다.

1. 표지
2. 핵심 긴장 또는 질문
3. 오늘의 흐름
4. 배경·개념 섹션 구분
5. 핵심 개념의 구성요소와 관계
6. 핵심 가치
7. 기존 접근의 병목
8. 작동 원리 섹션 구분
9. 이전/이후의 전환
10. 강점과 한계
11. 입력–변환–검증–결과 경로
12. 운영 구조 섹션 구분
13. 구성요소와 사람의 역할 분담
14. 검증과 피드백 루프
15. 핵심 근거
16. 적용·판단 섹션 구분
17. 설계 원칙
18. 대표 사례
19. 도입 판단 기준
20. 대안 비교
21. 확인 질문
22. 세 문장 요약과 Q&A

내용에 맞지 않는 장은 합치거나 제거할 수 있다. 다만 `문제 → 개념 → 메커니즘 → 운영 구조 → 사례 → 판단 → 요약`의 논리와 첫 3장, 마지막 2장은 유지한다. 여러 챕터라면 이 흐름을 챕터별로 반복하기보다 하나의 핵심 질문 아래 재구성한다.

## 핵심 원칙

- 배경은 따뜻한 회백색, 액센트는 진청 한 색을 기본으로 사용한다.
- 한 슬라이드에는 하나의 메시지만 두고 제목, 한 줄 설명, 본문, takeaway 순서로 읽히게 한다.
- 제목은 사실 이름보다 결론형 문장으로 쓴다. `기술 A의 특징`보다 `기술 A는 검증 비용을 줄인다`가 낫다.
- 기본 글자 크기를 줄여 내용을 밀어 넣지 않는다. 내용이 넘치면 슬라이드를 나눈다.
- 표지, 섹션, 설명, 카드·KPI, 관계도, 흐름, 비교표, 사례, 판단, 요약 레이아웃을 내용에 맞게 선택한다.
- 외부 폰트, CDN, 프레임워크 없이 발표 폴더만으로 동작해야 한다.
- 템플릿 CSS와 JavaScript는 발표 폴더에 복사한다. 과거 발표에 일괄 덮어쓰지 않는다.

## 제공 컴포넌트

- `.slide--cover`, `.slide--section`, `.slide--closing`: 표지·장 구분·마무리
- `.two-col`, `.panel`: 문제/해법, 이전/이후, 두 대안 설명
- `.card-grid`, `.card`: 세 역할·가치·요점
- `.kpi-grid`, `.kpi`: 수치 또는 검증 가능한 세 근거
- `.flow`, `.access-path`, `.shift`, `.maturity-line`: 순서·책임 경로·전환·성숙 단계
- `.graph-scene`, `.graph-node`, `.graph-edge`, `.legend-row`: 개체와 관계의 개념도
- `.balance-grid`: 강점과 한계
- `.hybrid-loop`: 검증·피드백 운영 루프
- `.pillar-grid`: 네 설계 원칙
- `.use-case-grid`, `.conversation`: 사례의 역할 분담과 대화 흐름
- `.decision-grid`: 적합·비적합 신호
- `.compare-table`, `.semantic-grid`: 반복 필드 비교와 의미 구조
- `.question-list`: 발표 확인 질문
- `.code-block`, `.code-layout`: 핵심 코드·데모
- `.takeaway`, `.source-note`: 핵심 문장과 근접 출처
- `.deck-settings`, `.deck-settings-panel`, `.deck-quick-links`: 우측 상단 설정과 Index / Slides / Report 이동

## 리포트·슬라이드·목차 연결

- 리포트 게이트를 통과하기 전에는 생성된 구조 청사진을 회차별 슬라이드 본문으로 채우지 않는다.
- 리포트의 각 핵심 섹션은 1개의 섹션 구분 슬라이드와 2–4개의 메시지 슬라이드로 압축한다.
- 리포트 표·도형이 발표 화면에서 읽히면 같은 SVG·이미지를 직접 재사용한다. 복잡한 표·도형은 의미·번호·관계를 유지하며 발표에서 읽을 부분을 CSS/SVG로 재배치한다.
- 슬라이드의 `aria-label`은 화면 제목보다 짧고 구체적인 목차 항목이다. `deck.js`가 이 값을 사용해 목차를 자동 생성한다.
- 섹션 구분 슬라이드는 목차의 시각적 경계를 만든다. `aria-label="핵심 개념 섹션"`처럼 역할을 포함한다.
- 리포트와 슬라이드는 같은 용어·그림 번호·논리 순서를 쓰되 문장 밀도는 다르게 유지한다.
- 덱 `<main>`에는 `data-report-source="report.html"`, 모든 `.slide`에는 근거가 된 리포트 ID를 `data-report-refs`로 둔다.
- 리포트의 모든 `.report-section`과 `data-deck-use="required"` 그림을 적어도 한 번 참조한다. Report 링크는 가능하면 정확한 `report.html#<id>`로 연결한다.

## 편집 규칙

1. 기본 HTML의 안내 문장을 교재 근거와 발표자의 해설로 모두 교체한다.
2. 각 `<section class="slide">`에 내용을 설명하는 고유한 `aria-label`을 둔다.
3. 각 슬라이드의 주장·표·도형이 나온 리포트 절·표·그림 ID를 `data-report-refs`에 기록한다.
4. 이미지에는 구체적인 `alt`를 적고 회차 `assets/` 아래에 저장한다.
5. 인용, 데이터와 외부 이미지는 해당 슬라이드의 `.source-note`에 근접 출처를 표시한다.
6. 표는 반복되는 필드를 비교할 때만 사용하고, 본문 글자 크기를 줄여 표를 욱여넣지 않는다.
7. 기본 제공 키보드 이동, 자동 목차, 진행률, 우측 상단 설정 패널, 테마, 전체화면 코드를 제거하지 않는다.
8. 설정 패널의 `Index / Slides / Report` 링크와 마무리 슬라이드의 보고서·회차 목록 링크를 유지한다.

## 품질 기준

- 1280×720과 1366×768에서 제목·본문·footer가 겹치거나 잘리지 않는다.
- 390px 모바일에서 덱이 화면 안에 축소되고 버튼과 목차를 사용할 수 있다.
- 키보드 ←/→, Home/End, 목차 이동, 테마, 전체화면, 해시 URL이 동작한다.
- 우측 상단 설정 아이콘이 리포트와 같은 위치·형태로 보이고 `Index / Slides / Report` 링크가 모두 동작한다.
- 모든 일반 슬라이드에 결론형 제목과 takeaway가 있고, 근거 슬라이드에는 출처가 있다.
- 발표 자료만 읽어도 핵심 논리가 이어지며, 세부 근거는 `report.html`에서 찾을 수 있다.
- 모든 `data-report-refs`가 실제 리포트 ID이고, 모든 본문 절과 필수 그림이 발표자료에서 빠짐없이 다뤄진다.
- 인쇄/PDF에서 한 슬라이드가 한 페이지로 출력되고 제어 버튼은 숨겨진다.
