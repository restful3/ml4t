# 스터디 회차 산출물 청사진

이 문서는 Claude Code와 Codex가 같은 품질 기준으로 한 회차의 상세 리포트, 발표 슬라이드와 자동 목차를 만드는 공통 설계도다. 아직 ML4T study에 완성된 기준 회차가 없으므로, 이 청사진과 `study-report-v1`·`study-deck-v1` 템플릿의 완성 조건을 품질 기준으로 삼는다. 첫 회차를 이 수준으로 완성하면 그 회차가 이후의 구체적 기준 예시가 된다. 교재 내용을 복제하지 않고 논리를 같은 깊이로 재구성한다.

## 기준 예시

- 아직 ML4T study에 완성된 기준 회차가 없다. 첫 회차를 완성하기 전에는 이 청사진과 두 템플릿의 완성 조건을 기준으로 삼는다.
- 첫 완성 회차의 경로는 `docs/studies/machine-trading/presentations/<session-slug>/report.html`·`index.html`·`presentation.toml`가 되며, 이후 회차는 그 회차를 구체적 기준 예시로 참조한다.

기준 예시에서 재사용할 것은 색이나 문구가 아니라 다음 완성 조건이다.

- 리포트와 슬라이드가 서로 다른 읽기 목적을 가진 한 쌍이다.
- 원자료를 먼저 상세 리포트로 재구성하고, 검증을 마친 리포트를 유일한 발표자료 내용 원천으로 사용한다.
- 장의 전체 논리가 문제, 개념, 메커니즘, 운영 구조, 근거와 한계, 사례, 판단으로 이어진다.
- 반복되는 비교는 표로, 관계와 순서는 새로 구성한 SVG·HTML 도형으로 설명한다.
- 각 시각자료에는 번호, 제목, 대체 텍스트, 재구성 범위와 출처가 있다.
- 리포트와 슬라이드의 목차는 제목·`aria-label`에서 자동 생성된다.
- 모든 슬라이드는 근거 리포트 절·표·그림을 추적할 수 있고, 리포트의 핵심 SVG·표가 발표 화면에 실제로 반영된다.
- 브라우저, 모바일, 키보드, 인쇄/PDF에서 실제로 사용할 수 있다.

## 한 회차의 파일 계약

```text
docs/studies/<study-slug>/presentations/<session-slug>/
├── presentation.toml
├── report.html                 # 사후 학습용 상세 리포트
├── index.html                  # 발표용 슬라이드이자 공유 URL
└── assets/
    ├── figs/                   # 회차에 귀속되는 재구성 SVG·이미지
    ├── report.css
    ├── report.js
    ├── deck.css
    └── deck.js
```

새 회차는 반드시 `agent-support/scripts/new-presentation.py`로 만든다. 생성된 CSS와 JavaScript는 그 회차의 스냅샷이다. 공용 템플릿을 개선해도 과거 회차에 자동 덮어쓰지 않는다.

## 하나의 논리, 세 가지 표현

| 논리 역할 | 상세 리포트 | 발표 슬라이드 | 자동 목차의 이름 |
|---|---|---|---|
| 핵심 결론 | Section 00의 한 문장 결론·전체 논리표 | 핵심 질문, 오늘의 흐름 | `핵심 질문`, `오늘의 흐름` |
| 배경과 개념 | 정의, 경계, 구성요소, 기존 병목 | 섹션 구분 + 개념도 + 가치 + 병목 | `배경과 핵심 개념 섹션` 등 |
| 메커니즘 | 이전/이후, 입력–변환–검증–결과 | 전환 + 강점/한계 + 책임 경로 | `작동 원리 섹션` 등 |
| 운영 구조 | 역할 계약, 승인, 피드백·갱신·롤백 | 역할 분담 + 운영 루프 | `결합과 운영 구조 섹션` 등 |
| 근거와 한계 | 주장–근거–조건–실패 신호 | 핵심 근거 1–3개 | `주요 근거와 수치` |
| 사례 | 문제–역할–결과–실패 경계 | 대표 사례 1–2장 | `대표 적용 사례` |
| 판단과 다음 행동 | 적합/비적합 신호, 작은 실험, 중단 기준 | 판단표 + 대안 비교 | `도입 판단 기준`, `대안 기술 비교` |
| 정리 | 용어, 참고문헌, 토론 질문 | 확인 질문 + 세 문장 요약 | `발표 정리 질문`, `정리와 질의응답` |

목차를 별도 파일로 중복 관리하지 않는다. 리포트는 `.report-section`의 직접 자식 제목과 `h3`, 슬라이드는 각 `.slide`의 `aria-label`에서 JavaScript가 목차를 만든다. 목차가 어색하면 JavaScript를 고치기 전에 제목 구조와 `aria-label`을 고친다.

## 에이전트 작업 순서

### 1. 사실과 범위를 고정한다

1. `README.md`에서 날짜, 담당자, 챕터를 확인한다.
2. `agent-support/studies.toml`에서 `materials_path`, `slug`, 상태를 확인한다.
3. 활성 자료가 `source/`에 있을 것이라 가정하지 않는다. 종료된 책은 `archive/` 경로를 사용한다.
4. 교재의 실제 목차, 핵심 주장, 예제, 표·그림, 용어를 먼저 목록화한다.
5. 인용 가능한 원문과 발표자의 해설, 외부 검증 사실을 구분한다.

### 2. 상세 리포트를 먼저 설계하고 완성한다

작성 전에 아래 내용을 짧게 정한다.

- 발표 전체를 관통하는 질문 1개
- 발표 후 남길 결론 1개
- 리포트의 5–7개 본문 섹션
- 각 섹션을 대표할 표 또는 도형
- 발표에서 반드시 다룰 핵심 관계와 필수 시각자료
- 토론 질문 1–3개
- 확인이 필요한 외부 주장과 출처

이 단계에서는 회차별 슬라이드 본문을 쓰지 않는다. 원자료에서 확인한 주장·근거·예시·한계·용어·시각 관계를 먼저 독립적으로 읽히는 `report.html`에 완성한다. 모든 본문 절·표·그림에 안정적인 `id`를 붙이고, 발표에 반드시 사용할 그림에는 `data-deck-use="required"`를 둔다.

다음 조건을 모두 만족해야 리포트 게이트를 통과한다.

- 장의 핵심 논리와 실패 경계가 빈틈없이 설명된다.
- 표·그림에 번호, 제목, 대체 텍스트, 재구성 범위와 출처가 있다.
- 자동 목차와 전체화면 이미지 줌·팬·Esc 닫기가 동작한다.
- 1440px, 390px와 실제 A4 PDF에서 내용이 잘리지 않는다.
- 슬라이드에서 쓸 주장과 시각자료가 모두 리포트 안에 존재한다.

### 3. 시각자료를 리포트의 근거로 만든다

- 구조·관계·순서는 SVG 또는 HTML 도형으로 새로 구성한다.
- 반복 필드는 표로 비교한다.
- 장식만 하는 이미지와 의미 없는 KPI 카드는 만들지 않는다.
- 책의 그림을 그대로 베끼거나 캡처하지 않는다.
- 외부 수치와 이미지는 가까운 위치에 직접 출처를 표시한다.
- 한 챕터 리포트는 대체로 표·도형 6–10개를 목표로 한다. 내용상 불필요하면 숫자를 채우지 않는다.
- 리포트의 모든 `.report-figure`는 클릭·Enter로 전체 화면에서 열리고, 휠·버튼·핀치로 확대되므로 구체적인 캡션과 `alt`를 제공한다.

### 4. 검증된 리포트에서 슬라이드를 파생한다

리포트 한 핵심 주장에는 정의, 작동 방식, 예시, 한계, 판단 기준 가운데 적어도 두 가지가 따라야 한다. 짧은 카드와 목록만으로 본문을 끝내지 않는다. 리포트 게이트를 통과한 뒤에만 18–30장의 발표 흐름으로 압축한다.

발표자료에는 리포트에 없는 새 주장·수치·사례를 넣지 않는다. 필요하면 리포트를 먼저 고친 뒤 다시 파생한다. 리포트 SVG가 발표 화면에서 읽히면 동일 파일을 직접 사용하고, 복잡하면 핵심 관계와 그림 번호를 보존한 발표용 SVG/CSS로 재배치한다. 장식용 대체 그림으로 바꾸지 않는다.

추적 계약은 다음과 같다.

- `presentation.toml`: `workflow = "raw-report-deck-v1"`, `report_source = "report.html"`
- 덱 `<main>`: `data-report-source="report.html"`
- 모든 `.slide`: 실제 리포트 ID를 공백으로 나열한 `data-report-refs`
- 리포트의 모든 `.report-section`과 `data-deck-use="required"` 그림: 한 번 이상 슬라이드에서 참조
- 발표 화면의 Report 링크: 가능하면 `report.html#<id>`로 정확한 근거 위치에 연결

슬라이드는 다음 밀도를 지킨다.

- 결론형 제목 1개
- 왜 중요한지 설명하는 lead 1개
- 한 가지 관계·비교·흐름
- 청중이 기억할 takeaway 1개
- 필요한 경우에만 근접 출처 1개

### 5. 실제 화면으로 검증한다

최소 확인 화면은 다음과 같다.

- 리포트: 1440px 데스크톱, 390px 모바일, A4 PDF
- 슬라이드: 1280×720, 1366×768, 390px 모바일
- 키보드: 리포트 목차와 이미지 뷰어, 슬라이드 이전/다음·Home/End·목차·Esc
- 링크: Index / Report / Slides
- 오프라인: 저장소 루트의 로컬 HTTP 서버에서 외부 CDN 없이 동작

검증 명령:

```bash
python3 agent-support/scripts/build-index.py
python3 agent-support/scripts/build-index.py --check
python3 agent-support/scripts/validate-site.py --check-materials
```

## 완료 체크리스트

- [ ] `report.html`, `index.html`, `presentation.toml`, 로컬 자산이 모두 있다.
- [ ] 리포트가 슬라이드 문장 확장본이 아니라 독립적인 상세 자료다.
- [ ] 원자료 감사 뒤 리포트 게이트를 통과하고 나서 발표자료를 만들었다.
- [ ] 핵심 주장에 근거·조건·한계가 연결되어 있다.
- [ ] 표·그림에 번호, 제목, 대체 텍스트, 출처 또는 재구성 메모가 있다.
- [ ] 리포트 목차와 슬라이드 목차가 자동으로 생성되고 실제 내용 순서와 맞는다.
- [ ] 모든 슬라이드가 유효한 `data-report-refs`를 가지며 모든 본문 절과 필수 그림이 발표자료에서 다뤄진다.
- [ ] 리포트의 핵심 SVG·표·CSS 관계가 발표 화면에 직접 재사용되거나 의미를 보존해 재배치됐다.
- [ ] 이미지 전체화면·줌·팬·Esc 닫기와 포커스 복귀가 동작한다.
- [ ] 슬라이드가 16:9에서 잘리지 않고 키보드·전체화면·테마가 동작한다.
- [ ] A4 PDF에 과도한 빈 페이지, 잘린 표·도형, 중복 번호가 없다.
- [ ] `source/` 또는 `archive/` 원문 자산을 공개 HTML이 직접 참조하지 않는다.
- [ ] 빌드 인덱스와 사이트 검증이 통과한다.

## 참가자가 에이전트에게 요청하는 가장 짧은 문장

```text
나는 <날짜> <챕터> 발표 담당이야. 프로젝트의 study-presentation 스킬과 Chapter 1 완성 기준을 사용해 원자료를 먼저 상세 리포트로 완성·검증하고, 그 리포트의 내용·표·SVG를 적극 반영한 HTML 슬라이드와 자동 목차를 만든 뒤 실제 브라우저와 PDF까지 검증해줘.
```

게시까지 원할 때만 다음 문장을 덧붙인다.

```text
검증이 끝나면 내 변경만 커밋하고 PR을 만들어 병합한 뒤 GitHub Pages 배포까지 확인해줘.
```

## 아카이브 수명주기

책이 끝나면 학습자료는 `source/`에서 `archive/`로 이동할 수 있다. 에이전트는 `agent-support/studies.toml`의 `materials_path`만 따라가며, 공개 회차 경로 `docs/studies/<study-slug>/presentations/<session-slug>/`와 이미 공유된 URL은 바꾸지 않는다. 회차에 필요한 SVG·이미지·CSS·JavaScript는 모두 회차 폴더 안에 있으므로 원문 이동 뒤에도 발표와 리포트가 그대로 동작해야 한다.
