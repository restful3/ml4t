# 제5장. 실행 시스템 — 쉬운 해설판

> 이 글은 Ernest P. Chan의 *"Quantitative Trading"* (2판, 2021) 제5장의 전체 내용을 빠짐없이 담되, 전문 용어와 개념을 일상적인 비유와 풀어쓴 설명으로 재구성한 해설판입니다.

이 시점에서 여러분은 좋은 레시피(전략)를 개발하고 시식 테스트(백테스팅)를 마쳤으며, 식당을 열 장소(브로커리지)와 주방 장비(물리적 인프라)도 갖추었습니다. 이제 남은 것은 **주문을 받고 요리를 내보내는 시스템** — 즉 **자동화 거래 시스템(ATS, Automated Trading System)** 을 만드는 것입니다. 이 장은 그 시스템을 구축하는 방법과, 거래 비용과 기대 성과 사이의 괴리를 최소화하는 방법을 다룹니다.

---

## 자동화 거래 시스템이란?

**자동화 거래 시스템(ATS)** 은 다음 세 가지를 수행하는 소프트웨어입니다:

1. 브로커리지 또는 데이터 공급업체에서 **최신 시장 데이터를 가져오고**
2. 거래 알고리즘을 실행하여 **주문을 생성하며**
3. 그 주문을 브로커리지에 **자동으로 전송하여 체결**

때로는 이 모든 단계가 하나의 프로그램으로 완전히 자동화됩니다. 때로는 일부만 자동화되어 수동 단계가 필요합니다.

**비유로 설명하면 이렇습니다:**

식당에서 주문 시스템을 생각해 보세요. **완전 자동화 시스템** 은 키오스크로 주문을 받아 자동으로 주방에 전달하고, 조리 완료를 알리는 시스템입니다. **반자동화 시스템** 은 주문서를 프린트하면 직원이 주방에 들고 가는 방식이죠. 두 방식 모두 작동하지만, 고빈도로 주문이 몰리는 식당이라면 자동화가 필수입니다.

### 과거 vs 현재

완전 자동화 시스템은 과거에는 Java, C#, C++ 같은 고성능 언어를 다루는 전문 프로그래머가 필요했습니다. 하지만 현재는:

- **QuantConnect**, **Blueshift** 같은 플랫폼 — 아마추어 프로그래머도 가능 (Blueshift는 프로그래머가 아니어도 가능)
- **MATLAB Trading Toolbox**, **Python Backtrader**, **R IBroker** 같은 도구

### 가격 외 데이터의 필요성

어떤 시스템을 구축하든, 브로커가 제공하는 가격 데이터 외에 **추가 데이터** 가 필요할 수 있습니다:

- **실적 추정치(earnings estimates)**: Interactive Brokers에서 Zacks를 통해 무료 제공
- **배당 데이터**: 역시 무료
- **실적 발표 날짜/시간**: Wall Street Horizon을 통해 소액 유료

## 반자동화 거래 시스템 구축

![](images/7765c142b960ba826430716fb914fd379a3b8b14a7d269ab6221fb4bb21592e7.jpg)
FIGURE 5.1 반자동화 거래 시스템의 구조

**반자동화 거래 시스템(semiautomated trading system)** 에서는 사용자가 Excel, MATLAB, Python, R 같은 도구로 주문 목록을 생성합니다. 이 프로그램은 백테스트 프로그램과 동일한 경우가 많습니다 — 결국 백테스팅한 바로 그 전략을 실행하는 것이니까요.

### DDE 링크: Excel과 브로커를 연결하는 가장 간단한 방법

**DDE(Dynamic Data Exchange)** 링크는 Excel 셀에 삽입하는 표현식으로, 자동으로 실시간 데이터를 불러옵니다. 예를 들어:

```
=accountid|LAST!IBM
```

여기서 `LAST`는 최종 체결 가격, `IBM`은 종목 코드입니다. Interactive Brokers와 Goldman Sachs REDIPlus 등이 이 기능을 제공합니다.

### 주문 생성 워크플로

1. Excel 매크로나 MATLAB/Python/R 프로그램이 시장 데이터를 스캔
2. 거래 알고리즘을 실행하여 주문을 텍스트 파일로 출력
   - 각 줄은 `(종목, 방향, 수량)` 형식 — 예: `("IBM", "BUY", "100")`
   - Day Only, Good Till Cancel 등 부가 정보 포함
3. 이 텍스트 파일을 브로커의 **바스켓 트레이더(basket trader)** 또는 **스프레드 트레이더(spread trader)** 에 업로드하여 제출

**바스켓 트레이더(basket trader)** 는 여러 종목의 주문을 한 번에 업로드하여 원클릭으로 제출하는 도구입니다. **스프레드 트레이더(spread trader)** 는 페어 트레이딩처럼 여러 쌍의 증권에 대해 조건을 설정하고, 실시간 가격을 모니터링하며 조건 충족 시 자동으로 주문을 넣습니다.

### 저자의 실제 워크플로

**바스켓 트레이딩 (Interactive Brokers):**

1. 매일 개장 전 MATLAB 프로그램 실행 → 시장 데이터 수신, 알고리즘 실행
2. 1,000줄 이상의 주문 파일(1,000개 이상의 종목) 생성
3. BasketTrader에 업로드 → 원클릭으로 전체 제출
4. 일부는 개장 시 체결, 나머지는 이후 체결 또는 미체결
5. 장 마감 전 미체결 주문 일괄 취소 (버튼 한 번)
6. 기존 포지션 전체 청산 시에도 버튼 한 번

**스프레드 트레이딩 (REDIPlus):**

1. 개장 전 MATLAB로 페어별 지정가 생성 (개별 주식이 아닌 **스프레드에 대한 지정가** )
2. 스프레드 트레이더에 수동으로 지정가 입력 (또는 Excel 파일로 자동 업로드 가능)
3. 버튼 하나로 장 내내 자동 가격 모니터링 + 주문 실행 시작

### 반자동화의 적합한 상황

반자동화 시스템은 **하루에 몇 차례만** 주문을 생성하면 되는 경우에 적합합니다. 실시간으로 빈번하게 주문을 생성해야 한다면 DDE 링크나 Excel 매크로의 속도로는 부족합니다.

그러나 반자동화에는 한 가지 **장점** 이 있습니다: **주문이 브로커에 전송되기 전에 직접 확인(sanity check)할 수 있다** 는 것입니다. 완전 자동화의 위험을 보여주는 대표적 사례가 **Knight Capital Group** 입니다 — 2012년 8월 1일, 소프트웨어 오류로 **$4억 4천만 달러** 를 잃었습니다. (이 회사는 이후 고빈도 거래 회사 Virtu Financial에 합병되었습니다.)

## 완전 자동화 거래 시스템 구축

![](images/7d98efba793ab1e6b2fdbceccee8e225910a8caf965dfce754a80f192f22ebea.jpg)
FIGURE 5.2 완전 자동화 거래 시스템의 구조

**완전 자동화 거래 시스템(fully automated trading system)** 은 거래 알고리즘을 루프로 반복 실행하면서, 최신 가격을 지속적으로 스캔하고, 장중 내내 새로운 주문을 자동 생성하고 전송합니다. 아침에 "시작" 버튼, 장 마감 시 "종료" 버튼만 누르면 됩니다.

### 기술 요구사항

- 브로커가 **데이터 수신 + 주문 제출용 API** 를 제공해야 합니다
- API가 지원하는 언어(Visual Basic, Java, C#, C++)로 시스템을 작성하거나
- **QuantConnect**, **Blueshift** 같은 플랫폼 활용 (백테스팅 플랫폼에서 바로 실행 전환 가능)
- MATLAB의 **Trading Toolbox** 또는 undocumentedmatlab.com의 서드파티 툴킷
- **RESTful API** 를 제공하는 브로커(Alpaca, Interactive Brokers, Robinhood 등)는 어떤 언어든 HTTP Get/Post로 접근 가능 — 단, 지연 민감한 전략에는 부적합

### Excel로 완전 자동화가 가능할까?

이론적으로는 가능합니다. Excel 매크로에 루프를 만들어 DDE 링크로 데이터를 업데이트하고 주문을 제출하면 됩니다. 하지만 현실적으로:

- DDE 링크의 데이터 업데이트가 **느립니다**
- Interactive Brokers는 기본적으로 동시에 **100개 종목만** 업데이트를 허용합니다
- 주문 제출도 느려서, 실시간 시장 데이터에 반응하는 전략에는 **적합하지 않습니다**

### TradeStation 같은 올인원 플랫폼

TradeStation 같은 브로커는 백테스팅과 주문 제출을 하나의 플랫폼에서 제공합니다. 백테스팅한 프로그램을 그대로 실전 주문으로 전환할 수 있어 편리합니다. 하지만 단점은 MATLAB, Python, R 같은 범용 언어에 비해 **유연성이 떨어진다** 는 것입니다. 예를 들어, 주성분 분석(PCA)에 기반한 복잡한 전략(예제 7.4 참조)을 TradeStation에서 백테스팅하기는 상당히 어렵습니다.

## 프로그래밍 컨설턴트 고용하기

ATS를 직접 구축하기 어려운 경우, **프로그래밍 컨설턴트를 고용** 하는 것도 좋은 선택입니다. 특히 실행 속도가 중요한 고빈도 전략에서는 전문가의 도움이 큰 차이를 만듭니다.

### 비용은 얼마나?

- 시간당 $50\~$100
- 고정 요금: 프로젝트당 $1,000\~$5,000 (독립 트레이더의 대부분의 프로젝트)
- 브로커 API 경험이 있는 프로그래머를 브로커가 추천해줄 수 있음 (예: Interactive Brokers의 컨설턴트 웹페이지)
- elitetrader.com에서 검색 또는 의뢰 가능
- 최후의 수단: **Upwork.com** — 단, 금융 시장과 거래 기술에 대한 심층 지식이 부족할 수 있음

### 전략 비밀 보호 문제

프로그래머를 고용하면 전략이 유출될 걱정이 있습니다. **비밀유지계약(NDA)** 을 체결할 수 있지만, 프로그래머가 개인 계좌에서 전략을 실행하는 것을 탐지하기는 거의 불가능합니다.

저자가 제시하는 세 가지 대응 방법:

1. **현실 인정**: 대부분의 전략은 본인이 생각하는 만큼 독특하지 않습니다. 경험 많은 트레이더들이 이미 유사한 전략을 사용하고 있으므로, 프로그래머 한두 명 추가는 큰 영향이 아닙니다.

2. **대용량 전략의 경우**: 선물 거래 전략처럼 시장 영향이 작은 전략은 추가 참여자의 영향이 미미합니다.

3. **정보 분할(compartmentalization)**: 서로 다른 프로그래머에게 서로 다른 부분을 맡깁니다.
   - 프로그래머 A: 다양한 전략에 사용할 수 있는 **거래 인프라** 구축 (전략 내용은 모름)
   - 프로그래머 B: 입력 파라미터를 읽어들이는 **전략 로직** 구현 (실행 인프라는 없음)
   - 둘 다 **실제 파라미터 값** 은 모름

## 거래 비용 최소화

3장에서 거래 비용이 전략의 실제 수익에 미치는 영향을 보았습니다. 브로커를 바꾸는 것 외에도, 실행 방법에서 거래 비용을 줄이는 방법이 있습니다.

### 1. 저가 주식 회피

기관 투자자는 보통 **주가 $5 미만** 의 주식을 거래하지 않습니다. 이유:

- 동일한 자본으로 더 많은 주식을 사야 하므로 **총 수수료 증가**
- **매수-매도 스프레드가 비율적으로 더 넓어** 유동성 비용도 증가

### 2. 주문 크기 제한 (시장 충격 최소화)

**시장 충격(market impact)** 을 줄이려면 주문 크기를 유동성에 맞춰야 합니다.

- **핵심 규칙**: 각 주문이 **평균 일일 거래량의 1%를 초과하지 않도록** 합니다
- S&P 500 대형주에서는 이 한도에 도달하기 어렵지만, 소형주는 놀랄 만큼 유동성이 낮습니다

**구체적 예**: Bel Fuse Inc.(S&P 600 소형주 지수 편입 종목)의 3개월 평균 거래량은 약 30,000주, 주가 약 $10. 1%는 300주 = 겨우 $3,000. **지수에 편입된 종목도 이 수준** 인데, 편입되지 않은 종목은 더 심합니다.

### 3. 시가총액 기반 주문 크기 조절

시가총액이 수천만 달러에서 수천억 달러까지 **수 자릿수 차이** 나므로, 선형 비례 배분은 소형주에 사실상 0의 가중치를 부여하게 됩니다. 분산 투자의 이점을 유지하려면:

- 최대 대형주와 최소 소형주의 자본 가중치 비율을 **10배 이내** 로 제한
- **시가총액의 4제곱근에 비례** 하는 가중치를 사용하면 적절합니다

### 4. 주문 분할의 양날의 검

기관 투자자는 대규모 주문을 작은 주문으로 나눠 시간에 걸쳐 실행합니다. 시장 충격은 줄지만, **슬리피지(slippage)** — 신호 발생 시점 가격과 평균 체결 가격의 차이 — 가 커질 수 있습니다. 소규모 개인 트레이더에게는 주문 크기가 크지 않으므로 보통 이 방법이 **필요하지 않습니다.**

### 5. 브로커 실행 속도의 중요성

때로는 슬리피지가 본인의 통제 밖에 있습니다:

- 브로커 소프트웨어의 느린 주문 처리
- 리스크 점검 절차에 의한 지연
- 거래소 접근 파이프라인의 병목
- 충분히 깊지 않은 다크풀 유동성

이런 실행 비용 문제가 4장에서 다룬 **브로커 선택** 에 영향을 미치는 이유입니다.

## 페이퍼 트레이딩으로 시스템 테스트하기

ATS를 구축한 후에는 **페이퍼 트레이딩(paper trading, 모의 거래)** 으로 테스트하는 것이 필수입니다. 이것이 **실제 돈을 잃지 않으면서 소프트웨어 버그를 발견하는 거의 유일한 방법** 이기 때문입니다.

### 페이퍼 트레이딩의 핵심 이점

1. **선행편향(look-ahead bias) 발견**: 페이퍼 트레이딩을 시작하는 순간, 실시간으로는 특정 데이터를 주문 전에 얻을 수 없다는 사실을 깨달을 수 있습니다. 이 경우 "백지로 돌아가야" 합니다.

2. **ATS와 백테스트 비교 검증**: ATS가 생성한 거래와 손익을 백테스트 프로그램의 이론적 결과와 비교합니다. 차이가 거래 비용으로 설명되지 않으면 **소프트웨어 버그** 가 있을 가능성이 높습니다.

3. **전략에 대한 직관적 이해 획득**:
   - 손익의 변동성
   - 사용되는 자본 규모
   - 하루 거래 횟수
   - 데이터 준비에 걸리는 시간 등 운영상의 어려움

4. **운영 일정 최적화**: 저자의 경우, 매일 아침 데이터 다운로드와 파싱에 20분, 주문 전송에 15분이 걸렸습니다. 전략이 **35분 이내의 최신 데이터** 를 요구한다면, 실행 환경을 바꾸거나 전략을 수정해야 합니다. 이런 타이밍 문제는 페이퍼 트레이딩 전에는 파악하기 어렵습니다.

5. **데이터 스누핑 편향 발견**: 한 달 이상 페이퍼 트레이딩을 하면 진정한 **표본 외(out-of-sample) 테스트** 가 됩니다. 하지만 실전 거래가 시작되면 페이퍼 시스템에 주의를 덜 기울이게 되므로, 데이터 스누핑 편향은 보통 **소액 실전 거래를 시작한 후에야** 발견됩니다.

## 실전 성과가 기대에 미치지 못할 때

드디어 첫 주문이 체결되었습니다! 하지만 한 달, 두 달, 그리고 한 분기가 지나도 전략이 미미하거나 마이너스 수익을 보인다면? 이것은 신규 퀀트 트레이더에게 **흔한 경험** 입니다. 이때 체계적으로 원인을 진단해야 합니다.

### 진단 체크리스트 (단순한 것부터)

1. **ATS 소프트웨어에 버그가 있는가?**
2. **ATS가 생성한 거래가 백테스트와 일치하는가?**
3. **실행 비용이 예상보다 훨씬 높은가?**
4. **유동성이 낮은 주식을 거래하여 시장 충격이 큰가?**

실행 비용이 예상보다 높다면, 위의 거래 비용 최소화 섹션을 다시 읽어보는 것이 좋습니다.

### 두 가지 가장 무서운 원인

단순한 진단이 모두 해결된 후에도 성과가 나쁘다면, 두 가지 근본적 원인을 의심해야 합니다:

#### 1. 데이터 스누핑 편향 (Data-Snooping Bias)

전략에서 규칙과 파라미터를 **최대한 제거** 해 보세요. 백테스트 성과가 완전히 무너진다면, 데이터 스누핑 편향이 있을 가능성이 높고 **새로운 전략을 찾아야** 합니다. 백테스트 성과가 여전히 합리적이라면, 부진한 실전 성과는 **단순히 운이 나쁜 것** 일 수 있습니다.

#### 2. 레짐 시프트 (Regime Shift)

**레짐 시프트(regime shift)** 란 금융 시장 구조나 거시경제 환경이 급격히 변하여, 과거에 수익성이 있던 전략이 더 이상 수익을 내지 못하는 상황입니다.

최근 주목할 만한 두 가지 레짐 시프트:

**1) 주가 십진화 (Decimalization, 2001년)**

- 2001년 4월 이전: 미국 주가는 1/16, 1/18 단위로 호가
- 이후: 소수점 단위(센트)로 호가
- 영향: 가격 발견 과정의 마찰이 줄어들면서, **마찰과 비효율에서 수익을 얻는 통계적 차익거래의 수익성이 구조적으로 하락**
- 결과: 2001년 이전 백테스트 성과는 현재 성과보다 훨씬 높을 가능성

**2) 플러스 틱 규칙 폐지 (Plus-Tick Rule, 2007년)**

- 2007년 이전: 공매도는 "플러스 틱" 또는 "제로 플러스 틱"에서만 가능
- 2007년 6월: SEC가 이 규칙 폐지
- 2010년 2월: 대안적 업틱 규칙(Rule 201) 도입
- 영향: 공매도 전략의 백테스트에서 2007년 이전과 2010년 이후 기간은 **인위적으로 부풀려진 성과** 를 보일 수 있음. 유일하게 현실적인 백테스트 기간은 2007년 6월\~2010년 2월일 수 있습니다.

**공매도의 추가 문제 — "대차 어려움(hard to borrow)":**

플러스 틱 규칙과 별개로, 소형주나 저유동성 주식은 **공매도를 위한 주식 차입이 어려운** 경우가 많습니다. 아무도 주식을 빌려주지 않으면 공매도가 불가능하므로, 백테스트에서 수익성 높은 공매도 포지션이 **실제로는 불가능했을 수 있습니다.**

이 두 가지는 잘 알려진 레짐 시프트이지만, 특정 종목군에 적용되는 **더 미묘한 레짐 시프트** 도 있을 수 있습니다. 레짐 시프트를 자동으로 감지하는 모델은 7장에서 다룹니다.

## 요약 — 이 장의 핵심

### ATS의 세 가지 장점

1. 백테스팅된 전략에 **충실한 실행** 을 보장
2. 수동 작업 제거 — **여러 전략을 동시에** 운용 가능
3. **신속한 주문 전송** — 고빈도 전략에 필수

### 반자동화 vs 완전 자동화

| 항목 | 반자동화 | 완전 자동화 |
|------|---------|-----------|
| 주문 제출 | 텍스트 파일을 수동 업로드 + 버튼 클릭 | 프로그램이 자동으로 전송 |
| 프로그래밍 난이도 | 낮음 (Excel/MATLAB/Python/R) | 높음 (API 연동 필요) |
| 주문 사전 확인 | 가능 (인적 체크) | 불가능 (Knight Capital 사례 주의) |
| 적합한 전략 | 하루 몇 차례 주문 | 장중 실시간 반응 필요 |

### 실행 시 핵심 체크리스트

- 주문 크기가 평균 일일 거래량의 1%를 초과하지 않도록 제한
- 시가총액의 4제곱근에 비례하여 가중치 조절
- 페이퍼 트레이딩으로 버그, 편향, 운영 문제 발견
- 실전 성과 부진 시: 버그 -> 비용 -> 파라미터 단순화 -> 데이터 스누핑/레짐 시프트 순서로 진단

실전 성과가 부진하지만 **운이 나쁜 것이지 편향이나 레짐 시프트가 아니라고 믿는** 경우, 인내와 자본 보존이라는 상반된 요구를 어떻게 조화시킬 것인가? 이 중요한 문제는 다음 장(6장)에서 체계적인 자본 보존 방법을 통해 다룹니다.

## 핵심 용어 해설

| 용어 | 쉬운 설명 |
|------|----------|
| **자동화 거래 시스템(ATS)** | 시장 데이터 수신, 주문 생성, 주문 전송을 자동으로 수행하는 소프트웨어 |
| **반자동화 거래 시스템** | 주문 생성은 자동, 제출은 수동으로 하는 시스템 |
| **완전 자동화 거래 시스템** | 주문 생성부터 제출까지 전부 자동으로 수행하는 시스템 |
| **바스켓 트레이더(basket trader)** | 여러 종목의 주문을 한 번에 업로드하여 일괄 제출하는 도구 |
| **스프레드 트레이더(spread trader)** | 페어/스프레드 주문의 조건을 설정하고 실시간 모니터링하는 도구 |
| **DDE(Dynamic Data Exchange)** | Excel과 외부 애플리케이션 간 실시간 데이터를 교환하는 프로토콜 |
| **API(Application Programming Interface)** | 프로그램이 브로커와 데이터를 주고받고 주문을 전송하는 인터페이스 |
| **RESTful API** | HTTP 요청으로 접근하는 API. 언어 무관하지만 속도는 전용 API보다 느림 |
| **시장 충격(market impact)** | 본인의 주문이 시장 가격을 불리한 방향으로 움직이는 현상 |
| **슬리피지(slippage)** | 거래 신호 발생 가격과 실제 체결 가격의 차이 |
| **선행편향(look-ahead bias)** | 백테스트에서 실시간으로는 얻을 수 없는 미래 데이터를 사용하는 오류 |
| **데이터 스누핑 편향(data-snooping bias)** | 과도한 파라미터 최적화로 과거 데이터에만 맞는 전략을 만드는 오류 |
| **레짐 시프트(regime shift)** | 시장 구조나 거시경제 환경이 급격히 변하여 기존 전략이 무효화되는 현상 |
| **페이퍼 트레이딩(paper trading)** | 실제 돈 없이 가상으로 거래하여 시스템을 테스트하는 모의 거래 |
| **대차 어려움(hard to borrow)** | 공매도를 위해 주식을 빌리기 어려운 상태 |
